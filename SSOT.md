# ontoMeta SSOT

## 1. 文档目的

本文档是 ontoMeta 项目的 Single Source of Truth，用于统一描述本项目的目标、范围、核心概念、系统边界、关键流程与产品信息架构。后续产品设计、技术方案、数据建模、提示词设计、前后端实现均应以本文档为准。

---

## 2. 项目定位

ontoMeta 的目标是构建企业级本体建模系统。

系统通过本体建模理论，对企业内部业务的真实运行情况进行抽象、建模与表达，并将企业数据平台中的原始数据转化为具备业务语义、关系语义和业务逻辑语义的知识表达，最终形成能够反映企业真实业务运行状态的语义化数字孪生。

换言之，ontoMeta 并不直接替代企业现有数据平台，而是在现有元数据基础之上，完成“从数据到语义”的提升。

---

## 3. 系统边界

### 3.1 依赖基础平台

本系统架构于元数据管理平台 DataHub 之上。

企业的数据资产、元数据管理、数据域组织、表与字段等基础数据管理工作，由 DataHub 负责；ontoMeta 不重复建设这部分能力。

### 3.2 本系统专注范围

ontoMeta 专注于将 DataHub 中已有的数据资产与元数据，通过本体建模转化为业务语义。

其核心职责包括：

- 基于 DataHub 中的数据域、表、字段、主外键、事实与维度信息进行本体草稿生成
- 将数据结构映射为业务对象、属性、关系和业务逻辑
- 支持在工作区对本体草稿进行编辑、确认、发布与持续修订
- 形成能够表达企业业务真实运行情况的语义模型

### 3.3 DataHub 可获取的数据范围

根据 DataHub 开源文档，ontoMeta 可以从 DataHub 获取并使用的核心数据，主要包括以下几类：

#### 3.3.1 结构类元数据

- **Dataset / 数据集**：表、视图、主题对象等数据资产的基础实体
- **SchemaField / 字段**：数据集字段、列级结构与字段语义
- **Schema 历史与版本信息**：字段结构演进、schema 变更记录
- **Container / 容器**：承载数据集、目录、文件系统或更高层组织结构的实体

#### 3.3.2 组织与语义类元数据

- **Domain / 数据域**：业务分域、建模边界与语义上下文
- **GlossaryNode / GlossaryTerm / 业务术语**：业务词汇、术语层级、定义与解释
- **Tag / 标签**：可用于分类、治理、标注与语义补充
- **OwnershipType / Ownership / Owner**：所有者、责任人、责任团队等治理信息

#### 3.3.3 关系与血缘类元数据

- **Lineage / 血缘**：表到表、字段到字段、任务到数据集的上下游关系
- **Dataflow / Datajob**：数据流与数据任务，承载加工链路语义
- **SQL / 文件 / OpenLineage 等来源的自动血缘**：可从多种来源解析出的关系信息

#### 3.3.4 使用与质量类元数据

- **Usage / Query History**：访问、查询、热度、使用情况等行为数据
- **Profiling / 数据剖析**：列分布、空值、值域、统计特征等剖析结果
- **Assertions / Data Quality**：校验规则、质量断言、通过/失败状态
- **Freshness / 及时性**：数据新鲜度、延迟情况与时效语义

#### 3.3.5 展示与分析类元数据

- **Dashboard / 仪表板**：分析看板实体
- **Chart / 图表**：图表实体及其依赖对象

#### 3.3.6 机器学习类元数据

- **MLModel / 模型**：机器学习模型实体
- **MLModelGroup / 模型组**：模型集合与分组
- **MLModelDeployment / 模型部署**：部署环境与上线状态
- **MLFeatureTable / 特征表**：特征工程表
- **MLFeature / 特征**：单个特征定义
- **MLPrimaryKey / 主键**：特征或模型关联的主键语义

#### 3.3.7 规则与表单类元数据

- **Forms / 表单**：治理流程中的人工补充信息
- **Custom Assertions / 自定义断言**：针对业务场景补充的质量规则
- **Compliance / 审批与合规相关信息**：在数据治理流程中形成的人工决策记录

#### 3.3.8 可通过 API 读取的通用能力

- **GraphQL / OpenAPI**：统一查询与获取上述元数据
- **Ingestion 元数据**：各类来源、管道、采集配置与导入结果

上述范围说明：DataHub 为 ontoMeta 提供的是“元数据与治理语义输入”，而不是业务交易数据本身。ontoMeta 在建模时应优先使用这些数据来推导业务对象、属性、关系和业务逻辑，再由使用者在工作区编辑确认最终本体。

### 3.4 非目标

以下内容不属于 ontoMeta 的核心建设范围：

- 替代 DataHub 进行元数据采集、治理与存储
- 替代数仓、湖仓、主数据平台进行底层数据管理
- 直接承载业务事务系统功能
- 将本体生成完全自动化且跳过人工确认

## 4. 核心设计原则

### 4.1 DataHub 是元数据来源，ontoMeta 是语义建模层

DataHub 是企业数据资产与元数据管理平台，是 ontoMeta 的上游事实来源。

ontoMeta 以 DataHub 的数据域、表、字段及其关联关系为输入，对其进行业务语义提升，而不是重新定义底层元数据。

### 4.2 本体生成采用“LLM 草稿 + 人工编辑确认”机制

本体的生成必须分为两个阶段：

1. **LLM 预生成草稿**：系统基于 DataHub 元数据和相关上下文，生成初始本体草稿；
2. **人工编辑确认**：由使用者对草稿进行编辑、修订，并在重要操作前进行二次确认。

LLM 的职责是提高本体建模效率，人工的职责是保证业务正确性、可解释性和可落地性。

### 4.3 本体必须真实映射企业业务

生成与发布的本体，不应停留在“字段翻译”或“表结构描述”层面，而必须尽可能真实反映企业内部业务对象、业务关系与业务规则。

本体建模的目标是形成企业业务的数字孪生，使语义模型能够回答“企业业务是如何运转的”。

### 4.4 本体是系统的一等公民

系统中的所有建模、展示、编辑、审核与发布能力，都应围绕本体展开。

前端信息架构、后端领域模型、权限流程、版本机制、搜索与可视化能力，均应优先服务于本体的表达与管理。

## 5. 本体定义

在 ontoMeta 中，“本体”是对企业业务世界的结构化语义表达。它不是单一实体，而是由多个一等公民共同组成的完整语义系统。

本体由以下四类核心要素构成：

### 5.1 Object Type

**Object Type** 指业务对象，也可称为概念、类、对象。

它用于表达企业业务世界中稳定存在、可被识别、可被描述的事物，例如客户、商品、门店、员工、订单、合同等。

在数据平台中，Object Type 通常对应：

- 维表
- 主数据表
- 主题对象主表
- 其他能够稳定承载某类业务实体含义的数据表

Object Type 是本体建模的核心锚点，其他属性、关系和业务逻辑都围绕 Object Type 展开。

### 5.2 Property

**Property** 指业务对象的业务属性。

它用于描述 Object Type 的特征、状态、标识、分类与补充信息，例如客户名称、客户等级、商品品牌、员工入职日期等。

在数据平台中，Property 通常对应：

- 维表字段
- 主数据字段
- 对象描述性字段

Property 必须服务于业务语义表达，而不只是技术字段的原样搬运。一个字段只有在具备明确业务含义时，才应成为本体属性。

### 5.3 Relation Type

**Relation Type** 指不同业务对象之间的关系语义。

它用于描述对象与对象之间如何关联、依赖、从属、参与或作用，例如“客户下单订单”“订单包含商品”“员工属于部门”等。

在数据平台中，Relation Type 通常对应：

- 外键关系
- 桥表
- 事实表
- 其他体现对象关联语义的数据结构

Relation Type 的目标不是简单记录表连接方式，而是表达业务世界中的关系含义。

### 5.4 Business Logic

**Business Logic** 指业务逻辑。

它用于表达在业务语义层成立的规则、计算方式、判定方式和衍生逻辑，例如指标逻辑、标签逻辑、规则逻辑、口径定义等。

在数据平台中，Business Logic 通常对应：

- 指标逻辑
- 标签逻辑
- 规则引擎逻辑
- SQL / ETL / 代码中的业务计算规则

Business Logic 既可以关联到一个 Object Type，也可以关联多个 Object Type 与 Relation Type，是语义数字孪生中对“业务如何运作”的关键补充。

#### 5.4.1 绑定到表与字段

借鉴数仓中“指标和标签绑定到表/字段”的成熟做法，ontoMeta 把 Business Logic 与 ObjectType（表）、Property（字段）的依赖关系做成显式的一等绑定，而不是依赖命名文本模糊命中：

- **BusinessLogicObjectBinding**：业务逻辑 → 对象（表），带角色 `subject`/`dimension`/`output`
- **BusinessLogicPropertyBinding**：业务逻辑 → 属性（字段），带角色 `input`/`output`/`filter`/`group`

绑定来源分两种：

- `inferred`：草稿生成阶段由 LLM 或规则推断产出
- `manual`：人工在工作区编辑阶段新增或确认

绑定关系可被人工编辑（新增、修改 role、解除），并参与版本与变更日志。这样既能做“这条逻辑依赖哪些表与字段”的查询，也能反向回答“这张表/这个字段被哪些逻辑使用”，支撑影响分析与血缘追溯。

## 6. 本体与数据平台映射关系

ontoMeta 的核心任务，是将 DataHub 中的数据资产映射为本体语义。映射关系如下：

| DataHub / 数据平台中的对象 | ontoMeta 中的语义对象 |
| --- | --- |
| 数据域 | 建模工作区 / 语义上下文边界 |
| 维表、主数据表、对象主表 | Object Type |
| 字段 | Property |
| 外键、桥表、事实表 | Relation Type |
| 指标逻辑、标签逻辑、代码规则 | Business Logic |

需要强调的是，这种映射不是机械的一一翻译，而是语义解释过程。系统应允许 LLM 生成草稿，也允许人工在审核阶段对映射结果进行重命名、合并、拆分、删除、补充和纠偏。

## 7. 核心业务流程

### 7.1 上游读取

系统从 DataHub 读取数据域、表、字段及相关元数据信息，作为本体建模输入。

### 7.2 LLM 草稿生成

系统结合元数据、命名、结构关系和上下文信息，由 LLM 预生成本体草稿，包括：

- 业务对象识别
- 属性识别
- 关系识别
- 业务逻辑候选识别
- 候选命名与语义说明

### 7.3 LLM 生成阶段所使用的 DataHub 数据与本体映射

LLM 在生成本体草稿时，并不是笼统地“读取 DataHub 元数据”，而是会使用不同类型的数据去判断不同的本体组成部分。以下为推荐的输入映射。

| DataHub 输入数据 | 数据内容说明 | 主要用于本体部分 | 具体用途 |
| --- | --- | --- | --- |
| Domain | 数据域名称、定义、分域层级 | Object Type、Relation Type、Business Logic | 作为建模上下文边界，帮助 LLM 判断对象属于哪个业务域、关系是否跨域、业务逻辑是否属于该域核心能力 |
| Dataset 基本信息 | 表名、显示名、描述、平台、库表路径、类型 | Object Type、Business Logic | 判断一张表更像业务对象主表、维表、事实表、宽表还是规则结果表，为对象识别和逻辑识别提供第一层分类 |
| Container | 库、目录、文件夹、分层容器结构 | Object Type、Business Logic | 帮助理解数据所处层级，如 ODS、DWD、DWS、ADS、特征层、标签层、指标层，从而判断其更偏对象承载还是逻辑产物 |
| Dataset Schema | 字段列表、字段类型、字段顺序、主键候选字段 | Object Type、Property、Relation Type | 判断表的业务主题、识别主实体字段、识别描述性字段、识别关联键与交易键 |
| SchemaField 描述信息 | 字段中文名、英文名、注释、业务描述 | Property、Relation Type、Business Logic | 为属性命名、关系命名、逻辑口径推断提供最直接的语义依据 |
| 主键/唯一键候选 | 主键字段、唯一标识字段、编码字段 | Object Type、Property | 判断是否存在稳定业务实体，以及对象的身份标识属性是什么 |
| 外键关系 | 显式外键、引用目标表、引用目标字段 | Relation Type、Object Type | 识别对象之间的显式关系，帮助生成“谁关联谁”的候选关系语义 |
| 血缘 Lineage | 表级/字段级上下游、来源表、去向表 | Relation Type、Business Logic、Object Type | 用于识别派生对象、加工链路、事实与维度的组合关系，以及哪些逻辑是由多个对象共同产生 |
| Dataflow / Datajob | 任务名、调度链路、处理节点、任务说明 | Business Logic、Relation Type | 判断某段加工过程是否对应指标逻辑、标签逻辑、规则逻辑，补足单纯看表结构无法看出的业务计算语义 |
| SQL Queries / Query History | 查询语句、聚合逻辑、Join 方式、过滤条件、使用频次 | Business Logic、Relation Type、Property | 用于识别指标口径、标签规则、对象关联方式，以及哪些字段被稳定用作分组、筛选、统计 |
| Usage / 使用热度 | 被谁使用、访问频次、下游消费情况 | Object Type、Property、Business Logic | 帮助 LLM 判断哪些对象和字段是业务核心，哪些逻辑是真实被消费的核心语义，而不是冷数据或噪音 |
| Profiling / 数据剖析 | 值分布、空值率、枚举特征、数值分布 | Property、Object Type、Business Logic | 判断字段更像标识、分类、状态、金额、时间或标签结果字段，也可辅助识别枚举属性和规则结果 |
| Tags | 治理标签、主题标签、敏感标签、分类标签 | Object Type、Property、Business Logic | 作为语义补充信号，帮助判断对象类别、字段业务属性、逻辑敏感性和治理属性 |
| GlossaryTerm / GlossaryNode | 业务术语、定义、术语层级、同义表达 | Object Type、Property、Relation Type、Business Logic | 用于业务命名标准化，使草稿中的对象、属性、关系、逻辑尽量贴近企业内部统一术语体系 |
| Ownership / Owner | 负责人、归属团队、责任部门 | Object Type、Business Logic | 用于辅助判断对象归属的业务域和业务含义，尤其在名称模糊时帮助 LLM 借责任团队推断业务上下文 |
| Dashboard / Chart | 看板名称、图表名称、指标展示对象、依赖数据集 | Business Logic、Object Type、Property | 反向识别真正被业务消费的指标、维度和核心对象，帮助确定哪些逻辑应进入正式本体 |
| Assertions / Data Quality | 质量规则、断言条件、校验口径 | Business Logic、Property | 识别数据中被显式约束的业务规则、状态规则、阈值规则，补充业务逻辑候选 |
| Freshness | 更新频率、延迟情况、产出时效 | Business Logic、Object Type | 辅助区分主数据、交易数据、汇总结果和实时指标，并推断业务逻辑的运行节奏 |
| Forms / 人工表单补充 | 人工填写的治理说明、认证信息、补充语义 | Object Type、Property、Relation Type、Business Logic | 为 LLM 提供结构化但非技术字段的信息，补足仅凭技术元数据无法理解的业务背景 |
| Schema History | 字段新增、删除、重命名、结构演进 | Object Type、Property、Business Logic | 帮助判断对象与逻辑是否稳定，识别临时字段、历史遗留字段和持续演化的业务概念 |

#### 7.3.1 Object Type 主要依赖的数据

LLM 识别业务对象时，优先依赖以下 DataHub 数据：

- `Domain`：确定对象所在业务域
- `Dataset 基本信息`：判断表是否具备独立业务实体语义
- `Container`：判断表所处分层与业务角色
- `Dataset Schema`：识别主键、名称、状态、分类等实体特征
- `主键/唯一键候选`：确认对象是否具备稳定身份
- `GlossaryTerm`：将技术表名映射为业务对象命名
- `Ownership`：辅助判断对象归属的业务团队
- `Usage` 与 `Dashboard/Chart`：判断该对象是否为真实业务核心对象

结论上，Object Type 的生成最依赖“表级语义 + 实体结构 + 业务术语 + 使用场景”四类信息。

#### 7.3.2 Property 主要依赖的数据

LLM 识别属性时，优先依赖以下 DataHub 数据：

- `SchemaField` 与字段注释：判断字段业务含义
- `Dataset Schema`：理解字段在对象中的结构位置
- `Profiling`：判断字段是枚举、金额、时间、状态还是标识
- `GlossaryTerm`：将字段对齐到标准业务术语
- `Tags`：补充字段分类、敏感性与治理属性
- `Usage / Query History`：识别常用筛选维度、分组维度与展示属性
- `Assertions`：识别被显式约束的重要字段规则

结论上，Property 的生成最依赖“字段语义 + 字段统计特征 + 业务术语映射”。

#### 7.3.3 Relation Type 主要依赖的数据

LLM 识别关系时，优先依赖以下 DataHub 数据：

- `外键关系`：识别显式对象连接
- `Lineage`：识别隐式依赖和加工关系
- `Dataflow / Datajob`：识别通过任务形成的关系语义
- `SQL Queries`：识别 Join 逻辑、桥接逻辑、多表组合逻辑
- `Dataset Schema`：识别关联键、桥表键、事实表键
- `GlossaryTerm`：标准化关系名称与关系方向
- `Domain`：判断关系发生的业务上下文

结论上，Relation Type 的生成最依赖“连接结构 + 加工链路 + 业务命名”。

#### 7.3.4 Business Logic 主要依赖的数据

LLM 识别业务逻辑时，优先依赖以下 DataHub 数据：

- `SQL Queries / Query History`：识别指标公式、标签规则、过滤条件、窗口逻辑
- `Dataflow / Datajob`：识别任务级计算过程与产出逻辑
- `Lineage`：识别逻辑依赖了哪些对象、字段和中间结果
- `Dashboard / Chart`：识别实际对外展示和消费的业务指标
- `Assertions`：识别显式业务规则与质量规则
- `Profiling`：辅助判断逻辑输出字段的结果形态
- `GlossaryTerm`：对齐指标、标签、规则的标准命名
- `Forms`：补充人工维护的口径与治理信息
- `Freshness`：判断逻辑是实时、日批、月批还是离线汇总逻辑

结论上，Business Logic 的生成最依赖“计算表达 + 加工过程 + 消费场景 + 人工补充说明”。

#### 7.3.5 一份 DataHub 数据可能同时作用于多个本体部分

需要明确，LLM 生成时不是“一类输入只生成一类本体结果”，而是同一份 DataHub 数据会同时作用于多个本体部分。

例如：

- 一张事实表的 `Schema + Lineage + SQL`，既可能帮助识别 `Relation Type`，也可能帮助抽取 `Business Logic`
- 一个字段的 `注释 + Profiling + Usage`，既可能形成 `Property`，也可能暴露某个 `Business Logic` 的关键口径字段
- 一个 `GlossaryTerm` 既可能用于标准化 `Object Type` 名称，也可能用于约束 `Property`、`Relation Type` 和 `Business Logic` 的命名

因此，系统设计上应将 DataHub 输入视为可复用的多维证据，而不是单次单用途的数据映射。

### 7.4 人工审核与编辑

人工对草稿进行审核和编辑，修正 LLM 可能存在的偏差，包括但不限于：

- 业务命名不准确
- 对象边界定义不合理
- 属性与关系归属错误
- 业务逻辑口径不完整
- 无法反映真实业务流程

### 7.5 发布

审核通过后，本体进入发布状态，成为当前企业业务语义的正式表达。

### 7.6 持续演进

随着 DataHub 元数据变化、业务流程变化或认知深化，已发布本体应支持持续修订与版本演进。

## 8. 产品信息架构

系统菜单导航定义如下。

### 8.1 工作区

**定位**：面向建模作业过程的入口。

**职责**：

- 按数据域组织本体建模工作
- 提供预处理入口，由系统后台自动调取对应的 DataHub 数据
- 承接从上游元数据到本体草稿的转换流程
- 提供草稿结果的编辑、保存与继续处理能力
- 提供 DataHub 页面链接作为可选查看入口

**设计原则**：

- 工作区不重复展示 DataHub 原始页面内容
- 工作区强调“从数据域出发开展建模”
- 工作区以预处理与编辑操作为核心
- 工作区是建模流程视角，不是最终语义消费视角

### 8.2 本体

**定位**：面向语义结果的核心展示与管理中心。

**职责**：

- 展示业务对象及其之间的关系
- 展示单个业务对象的详细语义信息
- 在对象详情页中展示属性、关系、业务逻辑等完整本体信息
- 通过卡片、列表、图等形式表达本体结构

**展示对象**：

- Object Type
- Property
- Relation Type
- Business Logic

**设计原则**：

- 本体模块是系统核心中的核心
- 展示方式应优先支持理解业务语义，而不是仅展示技术结构
- 图谱、列表、卡片等视图只是表达手段，核心是准确表达对象与关系

### 8.3 业务逻辑

**定位**：面向规则、指标、标签等逻辑语义的独立视图。

**职责**：

- 展示指标、标签等业务逻辑及其计算规则
- 展示业务逻辑关联的表与对象
- 支持对来源于代码解析或人工补充的业务逻辑进行管理
- 业务逻辑模块仅提供只读展示，不提供编辑入口

**业务逻辑来源**：

- 已有代码解析
- 人工添加

**设计原则**：

- 业务逻辑不是附属信息，而是本体的重要组成部分
- 业务逻辑页面应体现“规则如何连接数据与业务语义”
- 应支持从业务逻辑追溯相关对象、属性、关系与底层数据来源

## 9. 角色分工

### 9.1 DataHub

负责企业数据资产和元数据管理，包括但不限于：

- 数据域
- 表
- 字段
- 技术血缘
- 元数据采集与维护

### 9.2 LLM

负责根据已有元数据和上下文进行高效率的本体草稿生成，包括候选对象、属性、关系和业务逻辑。

LLM 产出默认视为候选草稿，而非最终事实。

### 9.3 人工审核者

负责从业务真实性、语义准确性、企业理解一致性等维度，对本体进行审核、编辑和发布。

人工审核者是本体最终质量的责任主体。

### 9.4 ontoMeta 系统

负责连接 DataHub、LLM 和人工审核流程，承载本体的生成、展示、编辑、管理与发布。

## 10. 正确性标准

一个高质量的 ontoMeta 本体，应同时满足以下标准：

1. **来源清晰**：能够追溯到 DataHub 中的原始元数据来源；
2. **语义准确**：命名、定义、关系和逻辑符合企业业务认知；
3. **结构完整**：对象、属性、关系、业务逻辑表达完整；
4. **可审核**：每个本体要素都可被人工理解、编辑和确认；
5. **可演进**：能够随着业务和数据变化持续更新；
6. **可形成数字孪生**：能够较真实地映射企业业务运行方式，而不是停留在静态表结构层。

---

## 11. 关键产品结论

1. ontoMeta 不是元数据平台，而是构建在 DataHub 之上的语义建模系统；
2. ontoMeta 的核心价值是把“数据平台中的原始数据”提升为“关联业务的语义表达”；
3. 本体生成必须坚持“LLM 生成草稿 + 人工审核发布”的双阶段机制；
4. 本体是一等公民，Object Type、Property、Relation Type、Business Logic 是系统最核心的四类语义对象；
5. 系统最终目标不是做字段解释工具，而是构建能够真实映射企业业务的语义数字孪生。

---

## 12. 后续实现约束

后续所有产品与技术设计都应遵守以下约束：

- 不脱离 DataHub 单独定义元数据事实来源；
- 不将 LLM 结果直接视为最终发布结果；
- 不以技术表结构替代业务语义建模；
- 不弱化 Business Logic 在本体中的核心地位；
- 不偏离“真实反映企业业务运行情况”的最终目标。
